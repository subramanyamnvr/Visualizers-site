Big Idea

Distributed systems are tradeoff engines, not one clever architecture diagram.

The design work is repeatedly deciding what can be stale, what must be serialized, and how broken partial progress gets repaired.

A. Replicate

Copy state for durability and reads

Useful for resilience and scale, but it introduces lag and conflict questions.

B. Partition

Split the problem space

Shard by user, tenant, region, topic, or time to stop every node from doing all the work.

C. Coordinate

Agree only when worth it

Consensus and strong ordering solve real problems, but they are expensive enough to use deliberately.

Durability & Reads

Replication improves resilience, but stale replicas change what clients can safely assume.

Ask whether reads can tolerate lag and whether writes need leader coordination or multi-writer conflict handling.

Leader-followerSimple write path, easy read scaling, but followers may lag.common default
Multi-leaderBetter write locality, but conflict resolution becomes a first-class design issue.coordination-heavy
LeaderlessQuorum ideas can improve availability, but reconciliation and read repair get trickier.conflict-prone
Critical question: if a user writes data and reads immediately, what are you promising about read-your-writes behavior?
Horizontal Scale

Sharding is really about choosing the failure and hotspot boundary you are willing to live with.

A good shard key spreads work while keeping the hottest workflows local enough to stay practical.

Healthy shard keys

High-cardinality keys that spread traffic naturally.
Keys aligned with data access and tenancy boundaries.
Keys whose hotspots can be predicted before launch.

Common shard problems

Celebrity or hot-tenant concentration.
Cross-shard fanout for common read paths.
Rebalancing pain once the original key choice ages badly.

Asynchronous Work

Queues absorb load; streams preserve ordered event history.

Both decouple producers from consumers, but they support different replay, ordering, and retention expectations.

Queues are best when

You need background jobs, retries, and worker elasticity.
Messages are mostly handled once and then discarded.
Per-message acknowledgment matters more than long-term history.

Streams are best when

Multiple consumers need the event history.
Replay, ordered partitions, and retention windows matter.
You are building materialized views or event-driven analytics.
Coordination

Consensus exists for the places where disagreement would be dangerous enough to justify the cost.

Use it for leadership, config, metadata, and strongly ordered control paths, not for every fast data-plane request.

Leader electionChoose one active coordinator at a time.

Config storeReplicate small, critical state with strong agreement.
Log orderingSerialize control decisions that cannot diverge.
Resilience Discipline

Retries only help when the operation is safe to repeat and the system has room to recover.

Without idempotency, backoff, and deadlines, retries can convert one transient failure into a cascading outage.

Good retry design

Idempotency keys or deduplication for side-effecting work.
Exponential backoff with jitter to avoid synchronized storms.
Deadlines and circuit breaking so callers eventually give up cleanly.

Failure multipliers

Infinite retries against a saturated downstream.
No reconciliation after partial success.
Assuming at-least-once delivery means exactly-once outcomes.
State Lifecycle

Distributed state becomes easier to reason about when you separate authoritative data, derived views, and repair paths.

The same data can exist as source-of-truth records, replica copies, cache lines, and replayable events. The design question is which copy each layer is allowed to trust.

Where state belongs

Authoritative state lives in the store or leader that owns correctness for a key or shard.
Derived state belongs in caches, projections, and materialized views that can be rebuilt.
Ephemeral work belongs in queues, retry buffers, and worker inboxes that can be replayed safely.

How repair happens

Detect divergence with heartbeats, lag metrics, offsets, and user-visible anomalies.
Repair with replay, reconciliation, backfill, or rebalancing instead of ad hoc manual edits.
Contain damage with idempotency keys, dedupe checks, and bounded retries.
