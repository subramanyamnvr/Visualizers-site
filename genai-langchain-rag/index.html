Feature Map

LangChain is the glue layer between data, models, and tools

The overview page is easy to miss the first time because the power is in the composition. LangChain gives you reusable building blocks for loaders, prompts, retrievers, models, tools, and traces so you can keep each decision visible instead of burying it in one giant prompt.

Loaders and text splitters

Turn raw files into metadata-rich documents before any retrieval happens.

  • TextLoader and web loaders create the document boundary.
  • RecursiveCharacterTextSplitter preserves local meaning better than fixed-length slicing.
  • Chunk overlap keeps supporting sentences close enough to retrieve together.

Embeddings and vector stores

Convert content into searchable coordinates so semantic recall works beyond exact keywords.

  • OpenAIEmbeddings or another embedding model maps text to vectors.
  • FAISS is a quick local store for experimentation and notebook work.
  • Metadata filters keep the answer scoped to the right policy, product, or section.

Retrieval chains

Separate retrieval from generation so you can tune evidence quality without rewriting the app.

  • create_stuff_documents_chain shapes the final context-aware prompt.
  • create_retrieval_chain connects the retriever to the answer step.
  • Top-k, reranking, and hybrid search control what the model actually sees.

Tools, traces, and evaluation

LangChain is not just RAG. It also supports model tools, traceability, and iterative debugging.

  • @tool and bind_tools let a model call external helpers.
  • LangSmith traces show prompts, tool calls, retrieved chunks, and failures.
  • Evaluation becomes easier when the chain is assembled from named pieces.

Design choices

Why this RAG shape works in production

The LangChain docs make the same decisions easy to express: model interface, retriever, prompt contract, and tracing. The important part is that each layer can be tuned independently without changing the others.

Chunking and parsing

  • Split by meaning first, not by arbitrary file structure.
  • Keep source ids, section names, and offsets for citations.
  • Use chunk size and overlap to balance completeness and retrieval precision.
  • Preserve headings so the retriever can land on useful spans instead of random slices.

Retrieval and reranking

  • Dense retrieval is strong when wording varies; hybrid retrieval helps when exact terms matter.
  • Reranking pushes the best evidence to the top before the answer prompt sees it.
  • LangChain gives you building blocks, so retrieval policy stays explicit in code.
  • Metadata filters can scope answers to a product version, region, or document family.

Prompt and output rules

  • State the answer contract in the system message so the model knows the boundary.
  • Ask for citations or source ids when the answer must be auditable.
  • Keep temperature low if the chain should prefer fidelity over style.
  • Structured outputs become useful when the downstream app needs labels, JSON, or tickets.

The chain shape from the local notebook flow

This is the compact version of the notebook pattern: load docs, split them, build a FAISS retriever, and combine retrieval with a strict prompt that only answers from the supplied context. The second block shows how the same model can pick up tools when retrieval alone is not enough.

LangChain + FAISS

Retrieval chain with grounded response rules

Why this pattern matters

  • It keeps retrieval separate from generation, which makes debugging much easier.
  • The retriever can be tuned independently from the answer prompt.
  • It maps directly to the notebook flow in the local GenAI material.
  • It also keeps a clean boundary when you later add tools or routing logic.

Operational notes

  • Use LangSmith tracing when the chain starts failing in subtle ways.
  • Adjust chunk size and top-k before changing model temperature.
  • Keep citation rules strict when the answer must stay grounded.
  • Prompts should say what to do when the context is incomplete instead of guessing silently.
from langchain_community.document_loaders import TextLoader
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_community.vectorstores import FAISS
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain.chains.combine_documents import create_stuff_documents_chain
from langchain.chains import create_retrieval_chain

docs = TextLoader("./policies/refund_policy.txt").load()
chunks = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=80
).split_documents(docs)

vectorstore = FAISS.from_documents(chunks, OpenAIEmbeddings())
retriever = vectorstore.as_retriever(search_kwargs={"k": 4})

prompt = ChatPromptTemplate.from_messages([
    ("system", "Answer only from the supplied context and cite source ids."),
    ("human", "{input}\n\nContext:\n{context}")
])

llm = ChatOpenAI(model="gpt-4.1-mini", temperature=0)
document_chain = create_stuff_documents_chain(llm, prompt)
rag_chain = create_retrieval_chain(retriever, document_chain)

result = rag_chain.invoke({
    "input": "Can annual subscriptions be refunded after 14 days?"
})
print(result["answer"])

Tool-aware extension

  • @tool works best for small, deterministic helpers.
  • bind_tools lets the model choose when a helper is needed.
  • This is useful when retrieval covers most cases, but a live lookup is still needed for edge cases.

Notebook pattern

  • The local GenAI notebook starts with a simple tool binding before moving into more complex orchestration.
  • That progression is why the page treats RAG as a shape, not just a single code snippet.
from langchain_core.tools import tool
from langchain.chat_models import init_chat_model

@tool
def get_weather(location: str) -> str:
    """Return a short weather summary for a location."""
    return f"Weather check needed for {location}."

model = init_chat_model("gpt-4.1-mini")
tool_model = model.bind_tools([get_weather])

messages = [
    ("system", "Use tools when the user asks for current facts or external data."),
    ("human", "Do I need an umbrella in Bengaluru today?")
]

response = tool_model.invoke(messages)
print(response)
This follows the same idea as the RAG notebook: retrieve first, answer second.

Production

What makes a LangChain app trustworthy outside the notebook

A demo can get by with a single prompt and a vector store. A real app needs a retrieval policy, a trace trail, and explicit behavior when the evidence is weak.

Tracing and review

Use traces to inspect the exact chunks and prompt that produced the answer.

  • Follow the path from question to retriever to answer.
  • Inspect prompt text when the model is overconfident.
  • Use traces to compare one configuration against another.

Evaluation and tuning

Measure the chain on answer correctness, grounding, and refusal behavior.

  • Change chunking and top-k before changing the model.
  • Check whether missing answers are retrieval failures or prompt failures.
  • Use the same question set across variants to compare behavior consistently.

Fallback behavior

Explicitly say what to do when evidence is absent or contradictory.

  • Return "I don't know" when the docs do not support a claim.
  • Prefer asking a clarifying question over inventing a detail.
  • Route to a live tool only when the question truly needs current data.

Metadata discipline

Keep source names, versions, and offsets so citations remain meaningful.

  • Metadata gives you scope control and better auditability.
  • Source ids should map back to the original document quickly.
  • This is especially important for policy or support answers.

Failure Modes

Where a nice-looking RAG app quietly breaks

Chunk boundaries hide the answer

The answer is present in the corpus, but the split cut the relevant sentence away from its supporting context.

Retriever recall is too narrow

Top-k is too low, metadata filters are too strict, or the hybrid signal is missing when exact terms matter.

Prompt is not strict enough

The model sounds confident even when the evidence is weak, so citations become decoration instead of control.

Tracing is an afterthought

Without trace data, you cannot tell whether the failure came from the retriever, the prompt, or the model.
