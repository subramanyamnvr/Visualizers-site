01 / Mental Model

Three frameworks, one architecture conversation.

All three vendors organize around non-functional quality attributes, but they package them differently. Azure centers five pillars and expands through workloads and service guides. AWS turns the model into a six-pillar framework with Lenses, labs, and a review tool. Google Cloud combines pillars with cross-pillar perspectives and strong principle language.

Shared quality goals sit in the middle. Azure connects sustainability as dedicated workload guidance, while AWS and Google Cloud treat it as a first-class pillar.

What all three frameworks are doing

  • They turn architecture review into a repeatable operating practice instead of a one-time design deck.
  • They make non-functional requirements concrete enough to inspect, test, and prioritize.
  • They force tradeoffs into the open so teams can justify why a workload is optimized the way it is.

Where they differ

  • Azure leans into design areas, workload guides, and staged maturity models.
  • AWS adds a broad ecosystem of Lenses, prescriptive guidance, labs, and the Well-Architected Tool.
  • Google Cloud adds perspectives, explicit core principles, and cross-pillar guidance for AI and regulated industries.

02 / Crosswalk

How the pillars line up across vendors.

The names are close enough to create a common language, but the framing is not identical. The same word can imply a different review surface or a different set of supporting artifacts depending on the cloud.

Read this carefully: Azure keeps sustainability as workload guidance connected to the five pillars, while AWS and Google Cloud elevate sustainability into the main framework. That changes how visible the topic is during routine reviews.

Operations, security, reliability, cost, and performance are universal structure. Sustainability is where the framework shape clearly splits.

1. Learn the common language

Use the crosswalk before comparing services. The pillars are the real vocabulary underneath service catalogs.

2. Review through workload shape

Availability targets, regulatory posture, demand variability, and spend constraints decide which pillar goes first.

3. Re-run the loop

As workloads grow, revisit targets, postmortems, cost signals, and operational evidence instead of treating architecture as frozen.

Tip: when a team says “make it well architected,” translate that into a pillar, a measurable target, a review artifact, and an operating habit.
