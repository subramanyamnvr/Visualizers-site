Study Frame

Databases are storage engines plus guarantees plus access patterns.

A lot of confusion disappears once you stop separating SQL syntax from physical access, locking, and workload design.

A. Schema

Defines business truth

Tables, keys, nullability, and constraints state what combinations of data should be possible.

B. Access path

Controls cost

Sequential scans, index seeks, covering indexes, and join order are how the database really does the work.

C. Workload

Changes everything

Transactional systems optimize for small, concurrent writes while analytical systems optimize for broad reads and aggregation.

Relational Core

Keys and constraints are part of your correctness model.

Think of tables as governed entities with relationship rules, not as loose buckets of rows.

Primary keyDefines identity and stable row uniqueness.
Foreign keyProtects relationships across entities.
Unique constraintEnforces business uniqueness outside the primary key.
Check constraintGuards allowed value ranges and invariants.
Practical rule: put correctness in the database when violating it would create expensive cleanup or hard-to-reproduce application bugs.
Performance Lever

An index helps when it matches the filter, sort, or join pattern you actually run.

Indexes trade write cost and storage for faster reads. The trick is choosing the access paths worth paying for.

Good index fits

High-selectivity filters on columns you search repeatedly.
Composite indexes that match the left-to-right filter and sort pattern.
Covering indexes that avoid extra heap lookups for hot queries.

Common misses

Indexing every column without tracking write overhead.
Building a composite index in the wrong column order.
Assuming an index helps when the query still touches most rows.

Execution Logic

Joins become expensive when row counts or cardinality assumptions explode.

Understand the planner well enough to ask which table is driving the query, how rows are reduced, and where sorting or spilling appears.

Nested loopGood when one side is tiny and the other has an efficient lookup path.index-friendly
Hash joinGood for larger equi-joins when enough memory exists to build the hash side.memory-heavy
Merge joinUseful when both sides are sorted or sortable on the join key.order-aware
Question to ask: where does the row set shrink? Early filters and sensible join order often matter more than a clever SQL rewrite.
Correctness Under Concurrency

Transactions are the database answer to race conditions and half-finished business operations.

Isolation levels are tradeoffs between concurrency and how much anomalous behavior you are willing to tolerate.

Core ACID ideas

Atomicity: either the whole business action commits or none of it does.
Consistency: constraints still hold after the transaction.
Isolation: concurrent work should not create invalid intermediate views.
Durability: committed changes survive crashes.

Typical anomalies

Lost updates when two requests overwrite each other.
Dirty reads when uncommitted work leaks into another session.
Phantoms when repeated reads see new qualifying rows.
Schema Design

Normalization reduces duplication; denormalization buys read convenience.

The right design depends on how expensive inconsistency would be and how often a query needs a pre-joined view.

Normalize when

You need strict updates, clean ownership, and minimal conflicting copies of the same business fact.

Denormalize when

Read performance or analytical convenience matters more than perfectly minimizing duplication.

Document assumptions

If duplicates exist, name the source of truth and how downstream copies are refreshed.

Workload Shape

OLTP and OLAP optimize for different pain points.

Trying to make one system do both perfectly usually creates tension around storage format, indexing, and concurrency.

DimensionOLTPOLAP
Primary jobServe transactional reads and writesAggregate, scan, and analyze large volumes
Typical queryFew rows, low latency, high concurrencyMany rows, wider scans, longer-running
Schema tendencyNormalized and constrainedStar, snowflake, or wide modeled tables
Storage biasRow-orientedOften columnar
Decision shortcut: if the system is about business state changes, think OLTP first. If it is about cross-cutting analysis over history, think OLAP first.
Execution Map

A good query plan is usually a story about selectivity, cardinality, and where the database can stop doing work early.

When you read a plan, you are trying to understand whether the engine can use an index, how many rows each step expects, and where memory or disk spill might appear.

How to read a plan

Find the driving table or access path first; it often explains most of the cost.
Check where row counts shrink, because early filters usually matter more than late rewrites.
Look for sorts, spills, or large hash builds that hint at memory pressure.

What plan shape tells you

A scan-heavy plan usually means the predicate is not selective enough for the chosen index.
A hash join often says the engine values set-building over repeated lookups.
A merge join usually means sorted access paths are being exploited well.
