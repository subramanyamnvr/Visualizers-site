API Mindset

Backend systems are communication contracts with operational consequences.

The useful question is not only how to expose functionality, but how the interface behaves under retries, abuse, version drift, and permission boundaries.

A. Interface

Shape the contract

Resource naming, status codes, schemas, and error models set developer expectations.

B. Identity

Control access

User auth, service auth, scopes, and session lifetime must align with real permissions.

C. Traffic

Protect the system

Limits, caches, and idempotency stop healthy usage from turning into accidental overload.

Transport Foundation

HTTP already contains more architecture than many teams realize.

Method semantics, headers, cache directives, and status codes are part of the product contract, not incidental wire details.

Method semantics`GET` should be safe, `PUT` idempotent, `POST` state-changing by default.
Status codesUse them to explain whether the client should retry, fix input, or ask for permission.
HeadersCarry auth, tracing, cache hints, content type, and version cues.
IdempotencyNecessary for safe retries on payment-like or create-like actions.
API Style Choice

Choose the interface style that fits the consumers, not the one that sounds more modern.

REST is broad and web-friendly; gRPC is strong for typed service-to-service communication and streaming.

DimensionRESTgRPC
Best fitPublic APIs, browser ecosystems, broad interoperabilityInternal service meshes, strongly typed contracts, streaming
PayloadUsually JSONUsually protobuf
DebuggabilityHuman-readable and easy to inspectMore tooling-dependent but compact and structured
Versioning tensionResource and field evolutionSchema compatibility and generated clients
Identity Flows

Auth flows exist to answer who is acting, on whose behalf, and with which scope.

Most confusion comes from mixing user auth, delegated access, and service identity into one vague token story.

Common user-facing flows

Session-based web login for first-party apps.
OAuth authorization code flow for delegated third-party access.
PKCE for public clients that cannot safely store a secret.

Service-facing flows

Client credentials for service-to-service trust.
Mutual TLS or workload identity inside infrastructure platforms.
Short-lived credentials to reduce leaked-secret blast radius.
State Strategy

Sessions and tokens solve different trust and revocation tradeoffs.

Do not choose solely by trend. Choose based on browser context, revocation needs, and who must validate the identity.

Server sessionsGreat for central revocation and browser apps, but require stateful storage.central control
Signed access tokensPortable and scalable, but revocation and over-broad lifetime need care.portable trust
Refresh tokensExtend user continuity while keeping access tokens shorter-lived.renewal layer
Rule of thumb: store only the minimum claims in tokens and treat authorization checks as a server-side concern, not only a frontend display concern.
Protection Layer

Rate limiting is how you express fairness and system protection in traffic terms.

The right policy depends on whether you are protecting a single endpoint, a user, an API key, or an entire tenant budget.

Fixed window

Simple and cheap, but bursty at the edges.

Sliding window

More accurate fairness, but usually more bookkeeping.

Token bucket

Good when you want controlled bursts with an average sustained rate.

Design question: what should happen after the limit is hit: hard reject, queued retry hint, or degraded response path?
Read Optimization

Caching is powerful because it changes the request path, but that also makes invalidation a correctness problem.

Use caches where reads repeat enough to justify the complexity, and be explicit about freshness guarantees.

Where caches appear

Browser and CDN caches for public or semi-public content.
Gateway or service caches for repeated computed responses.
Database query caches or materialized read models for expensive joins.

What breaks caches

Unclear ownership of invalidation triggers.
Personalized data stored without the right key boundaries.
Assuming stale data is harmless without checking the user journey.
