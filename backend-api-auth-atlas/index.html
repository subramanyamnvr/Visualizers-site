API Mindset

Backend systems are communication contracts with operational consequences.

The useful question is not only how to expose functionality, but how the interface behaves under retries, abuse, version drift, and permission boundaries.

A. Interface

Shape the contract

Resource naming, status codes, schemas, and error models set developer expectations.

B. Identity

Control access

User auth, service auth, scopes, and session lifetime must align with real permissions.

C. Traffic

Protect the system

Limits, caches, and idempotency stop healthy usage from turning into accidental overload.

02

TLS, CORS, WAF rules, coarse rate limits, request size limits, and trace IDs should happen before business logic.

03

The system validates the credential, loads the actor, checks tenant and scope, and rejects early when access is invalid.

Status Code Routing

Use status codes to tell clients their next move.

2xxRequest succeeded. Client can use the response. For async work, 202 should include a status resource or polling path.
3xxResource is elsewhere or cached copy can be reused. Redirect behavior matters for auth and method preservation.
4xxClient must change something: authenticate, fix validation, adjust scope, stop replaying, or wait after 429.
409The request may be valid, but current resource state conflicts with it. Version checks and duplicate keys live here.
5xxServer or dependency failed. Clients may retry only when the method and idempotency contract make replay safe.

Identity Flows

Auth flows exist to answer who is acting, on whose behalf, and with which scope.

Most confusion comes from mixing user auth, delegated access, and service identity into one vague token story.

Common user-facing flows

Session-based web login for first-party apps.
OAuth authorization code flow for delegated third-party access.
PKCE for public clients that cannot safely store a secret.

Service-facing flows

Client credentials for service-to-service trust.
Mutual TLS or workload identity inside infrastructure platforms.
Short-lived credentials to reduce leaked-secret blast radius.

Authorization code with PKCE

1
App creates verifier and challenge

The public client keeps the verifier locally and sends only the derived challenge.

2
User authenticates at identity provider

The app never sees the user's password. It receives an authorization code after consent.

3
App exchanges code plus verifier

The provider checks that the verifier matches the earlier challenge before issuing tokens.

4
API validates token and scopes

The backend checks issuer, audience, signature, expiry, tenant, and permission claims.

Flow selection guide

First-party web app: session cookie with server-side session storage is often simplest and easiest to revoke.
SPA or mobile app: authorization code with PKCE avoids storing a client secret in public code.
Service-to-service: client credentials, mTLS, or workload identity. No human user is present.
Third-party integration: OAuth scopes and consent boundaries must be visible and revocable.
Security boundary: authentication is not authorization. A valid token still needs a resource-level policy check.

State Strategy

Sessions and tokens solve different trust and revocation tradeoffs.

Do not choose solely by trend. Choose based on browser context, revocation needs, and who must validate the identity.

Server sessionsGreat for central revocation and browser apps, but require stateful storage.central control
Signed access tokensPortable and scalable, but revocation and over-broad lifetime need care.portable trust
Refresh tokensExtend user continuity while keeping access tokens shorter-lived.renewal layer
Rule of thumb: store only the minimum claims in tokens and treat authorization checks as a server-side concern, not only a frontend display concern.
JWT header {"alg":"RS256","typ":"JWT","kid":"key-2026-04"}
JWT payload {"iss":"https://identity.example.com","aud":"billing-api","sub":"user_914","tenant":"acme","scope":"invoice:read","exp":1776233600}
Policy decision allow only if token is valid, tenant matches route tenant, scope includes invoice:read, and user has access to invoice owner.

Store browser credentials carefully

Prefer HttpOnly, Secure, SameSite cookies for first-party browser sessions.
Avoid placing long-lived access tokens in localStorage for sensitive apps.
Protect state-changing requests from CSRF when cookies authenticate the user.

Keep claims small

Use tokens for stable identity facts, not constantly changing permissions.
Do resource-level authorization on the server for the specific object being accessed.
Use short access-token lifetimes when revocation speed matters.
Credential Lifecycle

Token systems need a complete lifecycle, not only a login endpoint.

1. IssueIdentity provider signs a short-lived access token with issuer, audience, subject, expiry, and scope.
2. PresentClient sends it to the API. The API validates it before reading business data.
3. AuthorizeServer combines claims with tenant, ownership, role, and resource state.
4. RefreshRefresh token renews access without keeping the access token long-lived.
5. RevokeLogout, password reset, device removal, or compromise must invalidate future access.
Protection Layer

Rate limiting is how you express fairness and system protection in traffic terms.

The right policy depends on whether you are protecting a single endpoint, a user, an API key, or an entire tenant budget.

Fixed window

Simple and cheap, but bursty at the edges.

Sliding window

More accurate fairness, but usually more bookkeeping.

Token bucket

Good when you want controlled bursts with an average sustained rate.

Design question: what should happen after the limit is hit: hard reject, queued retry hint, or degraded response path?

How to read the bucket

Refill rate is the steady budget. Burst capacity is stored credit. Incoming traffic spends tokens. When the bucket empties, the gateway rejects, queues, or degrades the request.

Read Optimization

Caching is powerful because it changes the request path, but that also makes invalidation a correctness problem.

Use caches where reads repeat enough to justify the complexity, and be explicit about freshness guarantees.

Where caches appear

Browser and CDN caches for public or semi-public content.
Gateway or service caches for repeated computed responses.
Database query caches or materialized read models for expensive joins.

What breaks caches

Unclear ownership of invalidation triggers.
Personalized data stored without the right key boundaries.
Assuming stale data is harmless without checking the user journey.
BrowserGood for static assets and user-local reuse. Dangerous for sensitive personalized data.
CDN / edgeGreat for public content, media, docs, and anonymous product data with clear TTLs.
GatewayCan cache shared API responses, enforce auth-aware keys, and absorb repeated reads.
Service / dataUseful for computed views and expensive joins, but invalidation ownership must be clear.

Retry Safety

Idempotency is how APIs survive duplicate attempts without duplicate business effects.

The network can fail after the server commits a change but before the client receives the response. Payment, order, booking, and invite endpoints need a replay story.

Payment create with idempotency key

1
Client sends unique key

`Idempotency-Key: pay_2026_04_15_abc123` travels with the create request.

2
Server checks dedupe store

If the key is new, reserve it before charging or committing the business action.

3
Action commits once

The final response is stored with the key after the transaction succeeds.

4
Retry returns same result

If the client retries, the server returns the original outcome instead of charging again.

Failure decisions

Same key, same payload: return the original response.
Same key, different payload: reject as a client conflict.
Key reserved, action still running: return pending or ask the client to retry later.
Key expired: document the retention window so clients know the risk.

The server should reserve the key before the dangerous side effect.

Production Review

A production API review should read like an incident-prevention checklist.

Before launch, review the API from the perspective of an SDK user, an attacker, an overloaded system, a future migration, and an on-call engineer debugging a bad request at 2 AM.

Endpoint names reflect stable product resources, not internal class names.
Every write endpoint has a documented retry and idempotency behavior.
Error responses include machine-readable codes and human-readable messages.
Auth checks validate signature, issuer, audience, expiry, tenant, scope, and resource policy.
Pagination is stable under concurrent inserts or documents the consistency tradeoff.
Rate limits are keyed by actor, tenant, API key, or endpoint according to the protected resource.
Cache keys include every dimension that can change the response.
PII and secrets are redacted from logs, traces, metrics, and client-visible errors.
Versioning rules explain additive fields, breaking changes, deprecations, and sunset dates.
Correlation IDs connect gateway logs, service logs, database queries, and downstream calls.

Client contract tests

Lock down status codes, required fields, optional fields, pagination, auth failures, and retry outcomes.

Abuse tests

Test missing auth, wrong tenant, huge bodies, fast retries, repeated idempotency keys, and malformed content types.

Observability tests

Confirm logs explain who called, what resource was touched, which policy ran, and why the response happened.

Final question: if this endpoint starts returning wrong data tomorrow, can the team identify whether the cause is contract drift, auth policy, cache keying, retry behavior, or a downstream dependency?

Control Plane

Strong API systems separate identity, policy, replay safety, and observability before a request reaches business logic.

The request path is easier to operate when auth, limits, cache keys, and traces are explicit controls rather than hidden middleware details.

Request controls

Authenticate the caller, then authorize the action at the tenant, scope, and object level.
Rate limit by actor or route so abuse and accidental storms do not share the same budget.
Attach trace IDs and idempotency keys so retries can be observed and deduplicated.

Response controls

Return status codes that tell the client whether to retry, fix input, or ask for permission.
Shape cache headers around the data's freshness and personalization needs.
Emit audit and metrics events so support teams can reconstruct what happened later.
